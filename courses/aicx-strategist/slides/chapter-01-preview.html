機械学習 → ディープラーニング → LLM

[図解推奨: ML→DL→LLMの包含関係をネスト図で]

機械学習（教師あり/教師なし/強化学習） ⊃ ディープラーニング（特徴量自動抽出） ⊃ LLM

  • 機械学習: データからパターンを学習。従来のAI（予測・識別）の多くはこの手法で開発
  • ディープラーニング: ニューラルネットを多層化。「特徴量の自動抽出」が最大のブレイクスルー
  • LLM（大規模言語モデル）: 「この文脈の次に来る最も自然な言葉」を確率的に予測する仕組み。予測の繰り返しで文章全体を生成。

★ 自律性の4段階（どこまで任せるか）

LEVEL 1
反応型
入力→即座に反応するだけ。Google翻訳のように、過去から学んだり自ら提案したりしない
LEVEL 2
支援型
AIが回答案を提示、人間が最終判断。安全だが工数削減効果は限定的
LEVEL 3
協働型
定型はAIが自動処理、複雑なものは人間にエスカレーション。実務での主流。工数大幅削減。
LEVEL 4
自律型
目標を与えれば計画〜実行まで完遂。技術的には可能だが判断ミスのリスクが大きい
実務の鉄則（公式テキスト）　支援型→協働型の段階的引き上げが基本。いきなり自律型を目指さない。「何を任せないか」の設計が同じくらい重要。

3つの起動タイプ（何がきっかけで動くか）

TYPE 1
指示型
人間がリアルタイムに指示を出すことで動作。チャットや質問入力がきっかけ。
例: ChatGPTにプロンプトを入力する
TYPE 2
定時型
あらかじめ設定した時間に自動で動作。導入効果が見えやすい（実行タイミングと処理内容が固定のため）。
例: 毎朝8時にニュースを要約・配信
TYPE 3
条件型
特定の出来事が発生したときに自動で動作。現場の通常業務フローをできるだけ変えずに組み込めるのが強み。
例: 在庫が一定数を下回ったら発注書生成
見分けるポイント　指示型 vs 条件型の区別：AIエージェントに直接依頼したかどうか。CRMへのデータ登録をきっかけに動くのは「条件型」（AIへの直接依頼ではない）

RPA vs AIエージェント

違いの本質：「思考や解釈を行う能力の有無」

観点RPAAIエージェント
処理対象定型業務（手順が完全に決まっている）非定型業務（文脈に応じた判断が必要）
入力形式構造化データが得意（CSVや定型帳票）非構造化データ（メール・チャット・自由記述）もOK
例外処理想定外の入力で処理が止まる文脈から推測して処理を続けられる

判断基準（3つ）　①例外的な判断が必要か？　②入力が定型か非構造化か？　③例外の発生頻度は？　→これで「定型処理 or LLM処理」を切り分ける

ハイパーオートメーション — 「いいとこ取り」設計

1つのプロセスの中に「定型ステップ」と「判断ステップ」が混在する → 最適な技術を割り当てる

STEP業務内容処理方式
1受注メールから添付ファイル（発注書）を取得定型処理（メールサーバー操作。判断不要）
2発注書の内容を読み取り、品目・数量・納期を抽出AI処理（取引先ごとに書式が違うためLLMが必要）

4受注管理システムに抽出データを登録定型処理（決まったフィールドへの入力）
5確認メールを送信定型処理

MCP（Model Context Protocol）とは

AIエージェントと外部ツールの接続を「共通の規格」で行えるようにする仕組み

2024年11月にAnthropicがオープン標準として公開。APIが「ツールごとに専用の窓口がある」のに対し、MCPは「どのツールも共通の窓口フォーマットに統一する」仕組み。

[図解推奨: AIエージェント ⇔ MCPサーバー ⇔ Slack / Google Drive / DB / etc. の3層構造]

AIエージェント ←MCP→ Slack MCP / Calendar MCP / DB MCP / ...

たとえると　USBの登場以前はプリンター用・カメラ用・キーボード用に端子がバラバラだった。USBが統一したように、MCPはAIの世界で同じことを実現しようとしている

MCPが解決する課題：M×N問題

BEFORE（個別API接続）
M×N通りの接続が必要
AIモデル3種×ツール5種 = 15通りの接続を個別に開発。AIもツールも増えるほど爆発的に増加。開発・保守コストが膨大。
AFTER（MCP）
M+N通りで済む
3種+5種 = 8通りの対応で完了。AIがMCPに対応 + ツールがMCPに対応するだけ。
+さらに強み：LLMが「今どんなツールが使えるか」を自動認識し、ツール選択もLLMが判断
出題予測

MCPの正式名称・提唱企業・目的をセットで。「将来の拡張コスト」への影響がストラテジストにとって重要な理由も問われる予測。

2026年以降の生成AI：3つの進化軸

トレンド 1
エージェント化
AIを「実行できる」ようにする。API連携・MCP・コンピュータブラウザ操作・フィジカルAIが支える潮流。AIの役割が「相談に答えること」→「業務を実行すること」へ
トレンド 2
ドメイン特化
AIを「専門家」にする。RAG・ファインチューニング・パーソナライズ・検索連携の4手段。汎用AI vs 特化型AIの使い分けが企業の競争力に直結
トレンド 3
モデルの強化
AIの「基礎能力」を上げる。モデル規模の拡大・マルチモーダル化・小型化・軽量化。エージェント化とドメイン特化の前提となる基盤能力の向上
出題予測

3つの進化軸の名称と内容の組み合わせ。「RAGはどのトレンドに分類されるか」→ドメイン特化。個々の技術用語よりも「何のためのトレンドか」で捉える。
